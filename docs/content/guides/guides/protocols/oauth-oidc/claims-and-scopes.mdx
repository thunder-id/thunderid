# Claims & Scopes

A **scope** is what a client *asks for*. A **claim** is what <ProductName /> *returns*. The mapping between the two is what controls which user attributes end up in your ID Token and your UserInfo response.

<ProductName /> implements three layers:

1. **Standard OIDC scopes** ([OIDC Core ┬¦5.4](https://openid.net/specs/openid-connect-core-1_0.html#ScopeClaims)) ŌĆö `openid`, `profile`, `email`, `phone`, `address` ŌĆö with fixed claim sets defined by the spec.
2. **Application-defined scope-to-claim mappings** (`scopeClaims`) ŌĆö request a scope, get a configurable bundle of claims.
3. **The `claims` request parameter** ([OIDC Core ┬¦5.5](https://openid.net/specs/openid-connect-core-1_0.html#ClaimsParameter)) ŌĆö request specific claims directly, claim-by-claim.

## Standard OIDC Scopes

| Scope | Claims returned |
|---|---|
| `openid` | `sub` (always included on any OIDC request) |
| `profile` | `name`, `family_name`, `given_name`, `middle_name`, `nickname`, `preferred_username`, `profile`, `picture`, `website`, `gender`, `birthdate`, `zoneinfo`, `locale`, `updated_at` |
| `email` | `email`, `email_verified` |
| `phone` | `phone_number`, `phone_number_verified` |
| `address` | `address` (structured JSON object) |

The actual values come from the authenticated user's attributes ŌĆö claims are omitted when the user has no value for them.

## Custom Scope-to-Claim Mapping (`scopeClaims`)

Each application can define its own scope-to-claim bundles. Asking for the scope returns the listed claims in both the ID Token and the UserInfo response.

```json
{
  "scopeClaims": {
    "profile": ["given_name", "family_name", "picture"],
    "email": ["email", "email_verified"],
    "roles": ["roles", "groups"]
  }
}
```

| Behavior | Detail |
|---|---|
| Overrides the standard mapping | Yes ŌĆö `profile` here returns only the three listed claims, not the full OIDC `profile` set |
| Custom scope names | Allowed ŌĆö `roles` here is an application-defined scope |
| Scope not in `scopeClaims` | Returns no extra claims (but the scope still passes through to access tokens) |
| Bypasses resource-server filtering | Yes ŌĆö see "Filtering rules" below |

## `claims` Request Parameter

Sometimes you want exactly one claim, not a whole scope's bundle. Pass a JSON object on the authorization request:

```http
GET /oauth2/authorize
  ?response_type=code
  &client_id=$CLIENT_ID
  &scope=openid
  &claims={"id_token":{"email":{"essential":true}},"userinfo":{"phone_number":null}}
```

| Sub-object | Targets |
|---|---|
| `claims.id_token` | Claims to include in the ID Token |
| `claims.userinfo` | Claims to include in the UserInfo response |

Inside each sub-object, claim names map to either `null` (request without constraint) or an object such as `{ "essential": true }` (request and mark as required).

<ProductName /> advertises `claims_parameter_supported: true` in [Server Metadata](../server-metadata).

## Filtering Rules

How the request-time scope list becomes the issued-token scope list:

| Scope category | Filtered by resource indicators? | Notes |
|---|---|---|
| `openid`, `profile`, `email`, `phone`, `address` | ŌØī Always kept | OIDC standard scopes are reserved and pass through unchanged |
| Custom scopes covered by `scopeClaims` | ŌØī Always kept | The application owns these |
| Permissions owned by a targeted resource server (see [Resource Indicators](../resource-indicators)) | Ō£ģ Kept | RS-defined |
| Permissions owned by a different resource server | ŌØī Dropped silently | RS-defined |
| Permissions not allowed on the application | ŌØī Rejected | Request fails with `invalid_scope` |
