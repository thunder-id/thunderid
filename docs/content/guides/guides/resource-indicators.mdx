# Resource Indicators

Resource indicators let a client specify which resource server an access token should target. When a client includes the `resource` parameter in an OAuth 2.0 request, <ProductName /> validates the request against a registered resource server. It filters the requested scopes to only those defined on that server and sets the access token's `aud` (audience) claim to the resource server's identifier. The receiving resource server can then verify the token by checking the `aud` claim.

This feature implements [RFC 8707 — Resource Indicators for OAuth 2.0](https://datatracker.ietf.org/doc/html/rfc8707).

## Prerequisites

- A running <ProductName /> instance.
- A registered application with OAuth 2.0 credentials (client ID and client secret for confidential clients).
- At least one resource server with:
  - An `identifier` set to an absolute URI (for example, `https://api.example.com/booking`). Resource servers with non-URI identifiers are not used in resource indicator flows.
  - Permissions (actions) defined on the resource server.
  - See [Manage Resource Servers](../resource-servers) for setup instructions.

## How Resource Indicators Work

Without resource indicators, an access token's audience defaults to the client ID. The token carries all scopes the client requested and was authorized for. Resource servers must trust the client to present the correct token to the correct service.

With resource indicators, the client declares the target resource server by passing its identifier in the `resource` parameter. <ProductName /> then:

1. Validates that the `resource` value is an absolute URI without a fragment component.
2. Resolves the URI to a registered resource server by matching against the `identifier` field.
3. Filters the requested scopes to only those defined as permissions on the targeted resource server. Scopes not defined on the resource server are silently dropped.
4. Sets the issued access token's `aud` claim to the resource server's identifier.

The resource server receiving the token verifies the `aud` claim matches its own identifier before accepting the request.

## Request an Audience-Restricted Token

### Client Credentials Grant

Use the `resource` parameter in a client credentials token request to target a specific resource server:

```bash
curl -kL -X POST https://localhost:8090/oauth2/token \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -u '<client-id>:<client-secret>' \
  -d 'grant_type=client_credentials' \
  -d 'scope=booking-api:reservations:create booking-api:reservations:view' \
  -d 'resource=https://api.example.com/booking'
```

**Response:**

```json
{
  "access_token": "eyJhbGciOiJSUzI1NiIs...",
  "token_type": "Bearer",
  "expires_in": 3600,
  "scope": "booking-api:reservations:create booking-api:reservations:view"
}
```

**Decoded access token claims:**

```json
{
  "sub": "<client-id>",
  "aud": "https://api.example.com/booking",
  "scope": "booking-api:reservations:create booking-api:reservations:view",
  "iss": "https://localhost:8090/oauth2/token",
  "exp": 1234567890,
  "iat": 1234564290
}
```

The `aud` claim contains the resource server's identifier. If any requested scopes are not defined on the targeted resource server, <ProductName /> silently removes them from the issued token. The response `scope` field reflects only the granted scopes.

### Authorization Code Grant

Include the `resource` parameter in the authorization request URL:

```
https://localhost:8090/oauth2/authorize?
  response_type=code&
  client_id=<client-id>&
  redirect_uri=https://yourapp.example.com/callback&
  scope=openid booking-api:reservations:view&
  resource=https://api.example.com/booking
```

<ProductName /> stores the resource indicator with the authorization code. At the token endpoint, the client can optionally narrow the resource set to a subset of what was authorized:

```bash
curl -kL -X POST https://localhost:8090/oauth2/token \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -u '<client-id>:<client-secret>' \
  -d 'grant_type=authorization_code' \
  -d 'code=<authorization-code>' \
  -d 'redirect_uri=https://yourapp.example.com/callback' \
  -d 'resource=https://api.example.com/booking'
```

Per [RFC 8707 Section 2](https://datatracker.ietf.org/doc/html/rfc8707#section-2), the `resource` values in the token request must be a subset of those in the authorization request. <ProductName /> rejects requests that attempt to expand beyond the originally authorized resources.

:::note
OpenID Connect scopes (`openid`, `profile`, `email`, `address`, `phone`) are not affected by resource indicator filtering. These scopes pass through regardless of which resource server the request targets.
:::

### Target Multiple Resource Servers

The `resource` parameter can appear multiple times in a single request to target more than one resource server. <ProductName /> resolves each identifier, filters scopes against each server's permissions, and combines the results.

```bash
curl -kL -X POST https://localhost:8090/oauth2/token \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -u '<client-id>:<client-secret>' \
  -d 'grant_type=client_credentials' \
  -d 'scope=booking-api:reservations:view payments-api:transactions:view' \
  -d 'resource=https://api.example.com/booking' \
  -d 'resource=https://api.example.com/payments'
```

**Decoded access token claims:**

```json
{
  "sub": "<client-id>",
  "aud": ["https://api.example.com/booking", "https://api.example.com/payments"],
  "scope": "booking-api:reservations:view payments-api:transactions:view",
  "iss": "https://localhost:8090/oauth2/token",
  "exp": 1234567890
}
```

When the token targets multiple resource servers, the `aud` claim becomes a JSON array containing each contributing server's identifier.

## Narrow the Audience on Token Refresh

When refreshing a token, the client can include the `resource` parameter to narrow the audience of the new access token to a subset of the original audience. This enables a client that originally obtained a broad token to request a more targeted token for a specific service.

```bash
curl -kL -X POST https://localhost:8090/oauth2/token \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -u '<client-id>:<client-secret>' \
  -d 'grant_type=refresh_token' \
  -d 'refresh_token=<refresh-token>' \
  -d 'resource=https://api.example.com/booking'
```

The requested `resource` values must be a subset of the original access token's audience. <ProductName /> drops any requested resource that does not appear in the original audience set. If the intersection is empty (none of the requested resources match the original audience), the request fails with an `invalid_target` error. Otherwise, the new access token's `aud` contains only the narrowed set.

:::note
Resource indicators on the refresh token grant can only narrow the audience. You cannot add resource servers that were not in the original token's audience.
:::

## Token Exchange with Resource Indicators

[Token exchange](https://datatracker.ietf.org/doc/html/rfc8693) (RFC 8693) supports both the `audience` and `resource` parameters simultaneously. The `audience` parameter carries opaque logical names, while the `resource` parameter carries resource server URIs that <ProductName /> resolves and validates.

```bash
curl -kL -X POST https://localhost:8090/oauth2/token \
  -H 'Content-Type: application/x-www-form-urlencoded' \
  -u '<client-id>:<client-secret>' \
  -d 'grant_type=urn:ietf:params:oauth:grant-type:token-exchange' \
  -d 'subject_token=<subject-token>' \
  -d 'subject_token_type=urn:ietf:params:oauth:token-type:access_token' \
  -d 'audience=logical-service-name' \
  -d 'resource=https://api.example.com/booking' \
  -d 'scope=booking-api:reservations:view'
```

<ProductName /> merges both into the issued token's `aud` claim. The `audience` values appear first, followed by the resolved resource server identifiers. Scope filtering applies only to the `resource` parameter because `audience` values are opaque and cannot be resolved to permission sets.

## How the Audience Claim Works

The `aud` claim in issued access tokens varies based on the request shape:

| Request Shape | `aud` Claim Result |
|---|---|
| `resource` targeting one resource server | Resource server identifier (string) |
| `resource` targeting multiple resource servers | JSON array of resource server identifiers |
| No `resource`, but scopes match registered resource servers | JSON array of contributing resource server identifiers |
| No `resource`, no matching resource servers | Client ID (string) |

When the `aud` claim contains a single value, <ProductName /> serializes it as a JSON string. When the claim contains two or more values, <ProductName /> serializes it as a JSON array. Both forms comply with [RFC 7519 Section 4.1.3](https://datatracker.ietf.org/doc/html/rfc7519#section-4.1.3).

## Validate Tokens at Your Resource Server

When your resource server receives a bearer token, validate the `aud` claim before processing the request:

1. Decode the JWT access token.
2. Extract the `aud` claim. Accept both string and array forms per RFC 7519 Section 4.1.3.
3. Verify that the `aud` value includes your resource server's identifier (for example, `https://api.example.com/booking`).
4. Reject tokens whose `aud` does not include your resource server's identifier.

```json
{
  "aud": "https://api.example.com/booking"
}
```

or

```json
{
  "aud": ["https://api.example.com/booking", "https://api.example.com/payments"]
}
```

Both forms confirm that the token is valid for `https://api.example.com/booking`.

## Error Responses

<ProductName /> returns `invalid_target` errors for resource indicator validation failures:

| Error Code | Condition |
|---|---|
| `invalid_target` | The `resource` value is not an absolute URI (missing scheme). |
| `invalid_target` | The `resource` value contains a URI fragment component. |
| `invalid_target` | The `resource` value does not match any registered resource server identifier. |

**Example error response:**

```json
{
  "error": "invalid_target",
  "error_description": "The target resource is invalid, unknown, or malformed."
}
```
