# Embed the OAuth and Flow Engine in Go

You can run the <ProductName /> OAuth 2.0/OIDC authorization server and flow execution runtime inside your own Go application. Import `github.com/thunder-id/thunderid/pkg/thunderidengine` and implement the host provider interfaces. <ProductName /> registers the runtime HTTP routes on your multiplexer.

## Prerequisites

Before you begin, ensure you have:

- Go 1.22 or later.
- A <ProductName /> server configuration tree (the same `repository/conf/deployment.yaml` and `repository/resources` layout used by `cmd/server`).
- Cryptographic key material referenced by your configuration (for example `repository/resources/security/crypto.key`).

## What the Engine Registers

The engine registers these runtime routes on your `http.ServeMux`:

| Route | Purpose |
|-------|---------|
| `GET /.well-known/openid-configuration` | OpenID Provider metadata |
| `GET /.well-known/oauth-authorization-server` | OAuth authorization server metadata |
| `GET /oauth2/jwks` | JSON Web Key Set |
| `GET /oauth2/authorize`, `POST /oauth2/auth/callback` | Authorization endpoint |
| `POST /oauth2/par` | Pushed Authorization Requests |
| `POST /oauth2/token` | Token endpoint |
| `POST /oauth2/introspect` | Token introspection |
| `GET` / `POST /oauth2/userinfo` | Userinfo |
| `POST /flow/execute` | Flow execution |
| `GET /flow/meta` | Flow metadata for Gate-style clients |

The engine does **not** register Dynamic Client Registration (`/oauth2/register`) or flow definition CRUD (`/flows/**`). The full <ProductName /> server keeps those routes.

## Implement Host Providers

Implement the interfaces in `pkg/thunderidengine/providers.go`:

| Provider | Role |
|----------|------|
| `ClientProvider` | Resolve OAuth clients, applications, and entity groups |
| `AuthnProvider` | Authenticate users and load attributes |
| `AuthzProvider` | Answer `IsAuthorized(subject, action, resource)` |
| `ResourceProvider` | Resolve resources by `resourceServerID/resourceID` URI |
| `OUProvider` | Resolve organization units and ancestors |
| `IDPProvider` | Resolve identity providers |
| `FlowDefinitionProvider` | Supply flow definitions for execution |
| `DesignProvider` | Resolve theme and layout JSON for `GET /flow/meta` |
| `I18nProvider` | Resolve translations and language list for `GET /flow/meta` |
| `RoleProvider` | Supply role names for `AuthAssertExecutor` token claims |
| `RuntimeStore` | Persist PAR, authorization codes, and flow context (required for host-only mode) |
| `ObservabilityProvider` | Optional event publishing |

`AuthzProvider` maps to internal permission checks as follows: each requested permission is checked with `IsAuthorized(entityOrGroupID, permission, "")`. Group membership uses the subject entity ID and group IDs from `ClientProvider.GetTransitiveEntityGroups`.

## Host-Only Bootstrap

By default, `Engine.Initialize` boots the full <ProductName /> domain stack and then replaces individual services when you supply providers. That is appropriate when you reuse <ProductName /> databases and only override a subset of behavior.

For a lightweight embed, set all required providers and enable **host-only** mode. Host-only initialization loads configuration and crypto (JWT/JWKS), wires your providers through bridges, registers only the executors you list, and **does not** start internal entity, role, theme, notification, or admin services.

Enable host-only mode explicitly:

```go
hostOnly := true
engine := thunderidengine.New(thunderidengine.EngineConfig{
    HostOnly: &hostOnly,
    // ...
})
```

Or omit `HostOnly` and supply every required provider; host-only mode is selected automatically when `ProvidersComplete` returns true.

Required providers for host-only:

| Provider | Required |
|----------|----------|
| `Client`, `Authn`, `Authz`, `Resource`, `OU`, `IDP`, `FlowDefinition` | Yes |
| `Design`, `I18n`, `Role`, `RuntimeStore` | Yes |
| `Observability` | No |

Set `HostOnly` to `false` when you want full internal bootstrap even though all providers are implemented (for example, gradual migration).

Host-only mode uses an in-process noop database transactioner for OAuth token and userinfo handlers when your `RuntimeStore` backs PAR, codes, and flow context. You still need a valid `deployment.yaml` and key material for JWT signing.

## Initialize the Engine

```go
import (
    "net/http"

    "github.com/thunder-id/thunderid/pkg/thunderidengine"
)

func main() {
    mux := http.NewServeMux()

    hostOnly := true
    engine := thunderidengine.New(thunderidengine.EngineConfig{
        ConfigPath: "/path/to/thunder/server/home",
        HostOnly:   &hostOnly,
        Providers: thunderidengine.Providers{
            Client:         myClientProvider,
            Authn:          myAuthnProvider,
            Authz:          myAuthzProvider,
            Resource:       myResourceProvider,
            OU:             myOUProvider,
            IDP:            myIDPProvider,
            FlowDefinition: myFlowProvider,
            Design:         myDesignProvider,
            I18n:           myI18nProvider,
            Role:           myRoleProvider,
            RuntimeStore:   myRuntimeStore,
        },
        Executors: thunderidengine.ExecutorConfig{
            Names: []string{
                "BasicAuthExecutor",
                "AuthAssertExecutor",
                "AuthorizationExecutor",
            },
        },
    })

    if err := engine.Initialize(mux); err != nil {
        panic(err)
    }

    // Listen on mux with your HTTP server.
}
```

Set `Executors.Names` to a subset of built-in executor names when you want a lean runtime. An empty `Names` slice registers the full default set (same as the <ProductName /> server). Use `InjectCustom` to register additional executors that implement `thunderidengine.ExecutorInterface`.

## Full <ProductName /> Server Integration

The <ProductName /> server (`cmd/server`) wires the same runtime routes through `enginebridge.RegisterServerRuntime` after it initializes domain services. Flow CRUD and DCR remain in `servicemanager` outside the engine surface.

## Custom Executors

Implement `thunderidengine.ExecutorInterface` (re-exported from the internal flow core) and pass instances through `ExecutorConfig.InjectCustom`, or supply `ExecutorConfig.CustomRegistry` for a fully custom registry.
